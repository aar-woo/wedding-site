<objective>
Make the DB the source of truth for the displayed guest name while keeping the
cached token name for instant render and graceful fallback.

Today `useGuestName` resolves the name synchronously from the `?t=` token (or
legacy `?to=`) with no network call. This plan adds an abort-safe DB fetch:
render the cached token name instantly, then replace it with the authoritative
`displayName` from `GET /api/guest/:id` on HTTP 200. Any failure (network, 404,
500, still-loading) silently keeps the cached name — the guest never sees an
error or a blank flash.

Purpose: DB becomes authoritative for the displayed name so edits to a guest's
display name take effect without re-issuing links.
Output: A pure testable fetch helper + its node:test, and an async-but-backward-compatible
`useGuestName` hook.
</objective>

  <action>
    Convert useGuestName to resolve the cached name synchronously (as today) and
    then override it from the DB via an abort-safe effect.

    1. Keep the existing synchronous resolution producing `tokenName`, `hasName`,
       and the decoded `id`:
       - Read `t = searchParams.get('t')`. If present, `decoded = decodeGuestToken(t)`.
         Set `tokenName = decoded.name`, `tokenId = decoded.id`, `hasName = true` when decoded.
       - Else legacy `?to=` path: trimmed value -> tokenName, hasName = true, tokenId = null.
       - Else tokenName = 'Our Beloved Guests', hasName = false, tokenId = null.
       (Preserve current precedence and wording exactly — D-03/D-04.)

    2. Add `const [dbName, setDbName] = useState(null);` for the DB override.

    3. The DISPLAYED name = `dbName ?? tokenName`. Compute as
       `const name = dbName ?? tokenName;`. This guarantees instant render of the
       cached token name (dbName starts null) and replacement once DB resolves.

    4. Reset + fetch effect keyed on `tokenId`:
       ```js
       useEffect(() => {
         setDbName(null);              // clear stale override when token/id changes
         if (!tokenId) return;         // legacy ?to= / no-token -> no fetch (req #5/#6)
         const controller = new AbortController();
         let active = true;
         fetchGuestDisplayName(tokenId, { signal: controller.signal })
           .then((displayName) => {
             if (active && displayName) setDbName(displayName);
           });
         return () => { active = false; controller.abort(); };
       }, [tokenId]);
       ```
       The `active` flag guards against out-of-order resolution / post-unmount
       writes; AbortController cancels the in-flight request. fetchGuestDisplayName
       never rejects, so no .catch is needed (but adding one is harmless).

    5. document.title effect: keep existing rule but key on the RESOLVED `name`:
       ```js
       useEffect(() => {
         if (hasName) document.title = `Save the Date – For ${name}`;
       }, [hasName, name]);
       ```
       This makes the title follow the token name first, then the DB name once it
       loads. (Preserve D-03: only set when hasName.)

    6. Return shape — MUST stay backward compatible. Return
       `{ name, hasName, resolved: dbName !== null }`. GuestGreeting only reads
       `name`; the extra `resolved` field is additive and optional.
